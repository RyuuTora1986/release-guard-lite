# Scope Rules — 产品边界与反膨胀

## 这个产品是什么

一个极窄的发版守门员 CLI 工具。读 diff → 跑检查 → 判风险 → 出报告 → 推 Discord。

## 这个产品不是什么

- 不是 Web 应用（不做 UI）
- 不是 SaaS 平台（不做账号、多租户）
- 不是数据库应用（不做持久化存储）
- 不是测试框架（不做测试编排）
- 不是 AI DevOps 平台（不做泛化 AI 能力）
- 不是自动修复工具（只报告，不动代码）

## 范围守门规则

在实现任何功能前，必须通过以下检查：

1. **MVP 检查**：这个功能是否在 5 件事之内（读 diff / 检测风险文件 / 执行检查 / 生成报告 / Discord 推送）？如果不是，拒绝。
2. **复杂度检查**：这个功能是否需要新增超过 1 个依赖？如果是，需要显式说明理由。
3. **架构检查**：这个功能是否需要引入新的基础设施（数据库、队列、认证）？如果是，拒绝。
4. **用户检查**：这个功能对 1-10 人小团队是否真正有用？还是只是"看起来专业"？

## 禁止列表

以下功能在第一版中绝对不做：

- Web UI / Dashboard
- 用户注册 / 登录 / API Key
- 数据库 / ORM / 数据持久化
- 多仓库管理
- 自动修复 / 自动 PR
- GitHub App / GitLab 集成
- 历史报告对比
- 插件系统
- 自定义模板引擎
- 国际化 / 多语言

## 膨胀信号

如果你发现自己在做以下事情，立即停下来：

- 创建超过 2 层的目录嵌套
- 编写"通用"工具函数
- 设计"可扩展"的接口
- 添加"以后可能用到"的配置项
- 讨论"下一阶段"的功能
